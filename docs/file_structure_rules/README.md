# How to use this workspace

> **What this is.** The project-specific operating guide for **clean-bmad-workspace** — the **AGY quick-start
> skeleton** Daniel clones to start a new project. It explains what the folders are, where your work goes, and
> the rules you follow every session. It is the customized, this-project version of the home-base
> **workspace-standard**. Start here; drop to the canonical `../../AGENTS.md` for the routing table and gates.
>
> **Keep it generic.** This repo carries the *stack* (FastAPI + Google ADK · Next.js + React + TypeScript ·
> Firebase) and the BMAD method — **not a product.** No domain specifics. Clone → rename → fill the
> placeholders → build.
>
> **The other file in `docs/`** (`workspace-standard.md`) is a **home-base reference snapshot** — the generic
> spec written from the mission-control point of view. Read it for the *why*; follow **this README** +
> `AGENTS.md` for the *how*.

---

## 1. The one-paragraph model

The **folder is the app; markdown is the program; the model becomes the agent.** You don't read the whole
tree — `AGENTS.md` is the brain, its routing table tells you the *few* files a task needs, and everything you
produce is saved as plain markdown in `_artifacts/` so the next session (any model) can pick it up. No
database, no hidden state. This repo is a **self-contained leaf workspace**: its own git repo, its own
artifact memory that travels with it, BMAD-driven. Cloned to seed a real project — so it models the standard,
it doesn't carry a product.

## 3. The rules you follow every session

1. **Plan-first (the artifacts gate).** Before touching any file outside `_artifacts/`: write an
   `implementation_plan.md` into a dated `_artifacts/` folder, present it, and **STOP for "approved."** Skip
   only for read-only asks and trivial one-liners. Full rule → `.agents/rules/artifacts-always-first.md`.
2. **Least-context loading.** Don't read the tree. Use the `AGENTS.md` §6 routing table — "for this task read
   X, skip Y, use skill Z." If what you need isn't here, **GO BACK** to `../../router.md`.
3. **Git — never commit/push yourself.** Hand Daniel the exact command in the walkthrough's "Your Actions."
   Full rule → `.agents/rules/git-policy.md`.
4. **`_my_resources/` is off-limits** unless Daniel says otherwise (§8 of `AGENTS.md`).
5. **Rules live in `.agents/` — no forks.** Project-specific hard-stops go in a local `constitution.project.md`,
   never by hand-editing a vendored generic rule. This is the anti-drift guard.
6. **Keep it generic.** This is the skeleton, not a product — no domain specifics, secrets, or product data.

## 4. Where your work goes (artifacts — project-local)

Artifacts go **where you work FROM.** Open this project as your working dir → everything lands **project-local**
in this repo's `_artifacts/`, so the history travels with the code (and with every clone made from it):

```
_artifacts/
  INDEX.md                           <- the ledger (one row per session)
  2026-06-27_<slug>/                 <- a random/standalone task
    implementation_plan.md · walkthrough.md · task-list.md
  epic-1/  epic-2/                   <- BMAD stories grouped by epic
    story-1.1/ …
  _archived/                         <- retired history
```

> The live "pick up / hand off" brief is **not** in `_artifacts/` — it's BMAD's
> `_bmad-output/active-context/active-context.md`. `_artifacts/` holds session *history*; the brief holds
> *current* state. (See `/boot-sprint-context` and `/update-sprint-context`, which read/write that file.)

- **Random task** → `_artifacts/<YYYY-MM-DD>_<slug>/`.
- **A BMAD story** → `_artifacts/<epic>/<story>/` (epic folder houses its stories — create it if missing).
- **Close every non-trivial task** with `walkthrough.md` (+ a "Your Actions" section: manual steps + the exact
  git command), a `task-list.md` snapshot, an `INDEX.md` row, and an updated
  `_bmad-output/active-context/active-context.md`.

> **Finding history:** sessions run *from inside this project* save here. Sessions run *from the home base*
> save to the home base's `_artifacts/clean-bmad-workspace/`. To reconstruct full history, check **both**.

## 5. How this differs from the home base (the customization)

The reference doc in `docs/` (`workspace-standard.md`) was written for **Sudo_Hatter_Command** (mission control /
the lobby). Same principles, different role — here's the translation:

| | Home base (`Sudo_Hatter_Command/`) | This project (`clean-bmad-workspace/`) |
|---|---|---|
| **Role** | the **lobby** — routes you *into* projects | a **leaf workspace** — the skeleton you clone to seed a real app repo |
| **Navigation** | owns `router.md`; routes down to projects | up-routes to `../../router.md`; no router of its own |
| **Artifacts** | `_artifacts/<project>/` buckets + `_artifacts/_main/` | **one project-local `_artifacts/`** that travels with the repo |
| **Extra infra** | `_system/` (workspace builder), `_routing-canary/` | none — it's an app skeleton, not a workspace factory |
| **What drives work** | routing & toolkit upkeep | **BMAD stories** (`_bmad/` + `_bmad-output/`) |
| **Standard** | authors the canonical `workspace-standard.md` | **consumes** a vendored copy at `docs/workspace-standard.md` |

Practical upshot: when a reference doc says "the lobby," "router.md," or "per-project bucket," read it as
"**this** repo," "`../../router.md`," and "**this** repo's project-local `_artifacts/`."

## 6. Upkeep — the going-forward loop

**End-of-task checklist** (before saying "done" on anything that changed files):
- ☐ `walkthrough.md` — what changed + real pasted test output + "Your Actions" (the exact git command)
- ☐ `task-list.md` — snapshot of the final todo list
- ☐ `_bmad-output/active-context/active-context.md` — updated (the hand-off)
- ☐ `_artifacts/INDEX.md` — one row appended
- ☐ `docs/repo-map.md` — updated **only if** a top-level folder or agent was added/removed
- ☐ story `.md` status **and** `sprint-status.yaml` synced (story work only)

**When the structure itself changes** (folders, the toolkit, routing): edit the master in `.agents/` and
re-sync — never hand-edit a vendored copy; refresh `docs/workspace-standard.md` from the home-base canonical;
update this README if the project's own layout or workflow changed.
